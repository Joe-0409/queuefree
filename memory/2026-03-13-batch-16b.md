# Batch 16B: 真实来源 Registry 冻结

**Date:** 2026-03-13  
**Status:** ✅ Complete

## Summary

Registry 冻结批次，将真实 product / address / current-user 来源所需的最小读取面登记进 registry。不改外部 API contract，只冻结内部读取真相源。

## Registry 更新

**文件：** `docs/registry/registry-baseline-v1.2.md`

**新增内容：**
1. **Table Field Registry** — 4 个表的冻结字段
2. **Source Ports 读取范围** — 3 个 ports 的读取边界
3. **内部决策** — 404 处理策略冻结

## Table Field Registry

### `users`
- `userId: string`
- `phoneMasked: string`（为 `/v1/me` 预留）

### `user_addresses`
- `addressId: string`
- `userId: string`

### `products`
- `productId: string`
- `isQueueEligible: boolean`

### `product_skus`
- `skuId: string`
- `productId: string`
- `priceMinor: number`
- `currencyCode: string`
- `maxQty: number`

## Source Ports 读取范围

### `CurrentUserSourcePort`
- reads `users.userId`

### `AddressBookSourcePort`
- reads `user_addresses.addressId`
- reads `user_addresses.userId`

### `CatalogReadSourcePort`
- reads `products.productId`
- reads `products.isQueueEligible`
- reads `product_skus.skuId`
- reads `product_skus.productId`
- reads `product_skus.priceMinor`
- reads `product_skus.currencyCode`
- reads `product_skus.maxQty`

## 内部决策冻结

- ✅ 当前认证用户在 API 请求上下文中解析为 `userId: string`
- ✅ 非本人地址在写链路中按 `404` 处理
- ✅ 非队列商品在写链路中按 `404` 处理
- ✅ 不可下单 SKU 在写链路中按 `404` 处理

**404 选择理由：** 与现有写接口 contract 一致（POST /v1/orders 当前冻结的是 400 / 404 / 500，没有 409）

## 设计要点

**最小读取面原则：**
- 只冻结当前写链路真正需要的字段
- 未包含 `products.title` / `user_addresses.isDefault` / `product_skus.stockStatus` 等
- 更稳健，后续读接口可扩展需再开 registry patch

**预留设计：**
- 保留 `users.phoneMasked` 即使当前写链路主要只需要 `userId`
- 为 `/v1/me` 的真实来源预留，避免补 patch
